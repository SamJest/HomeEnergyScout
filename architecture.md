# Architecture

## Approach
The project uses a clean static directory structure with shared front-end assets and page-level configuration through `window.pageConfig`.

## Why this approach
- easy to edit locally
- easy to hand off
- clean URLs through directory-based pages
- generator-friendly later without requiring a rebuild of the architecture now
- simple enough for a small utility site, but structured enough for scale

## Current structure
- `/index.html` - homepage
- `/running-costs/index.html` - running cost hub
- `/savings/index.html` - savings and payback hub
- `/sizing/index.html` - sizing hub
- `/compare/index.html` - comparison hub
- `/guides/index.html` - guide hub
- trust/legal pages in their own directories
- `/assets/css/styles.css` - shared design system
- `/assets/js/site.js` - shared metadata, shell rendering, and tracking hooks

## Shared shell
The launch build now uses a prerendered shell generated by `scripts/prepare-static-launch.mjs`, with `assets/js/site.js` acting as the runtime enhancement and fallback layer.

That shared layer covers:
- canonical, Open Graph and Twitter defaults
- prerendered header, footer and breadcrumbs
- prerendered base schema and FAQ extraction
- calculator behaviour and result rendering
- event-tracking hooks for internal links, FAQs, calculators and future ad slots

## Batch 2 architecture decisions
- The main navigation now points directly to live hubs rather than trust-only pages.
- Footer structure now reinforces the main section architecture instead of acting as a generic legal footer only.
- Each hub is designed as an “entry and routing page” that can later evolve into deeper category landing pages without changing URL structure.
- Supporting pages are still to be added beneath these hubs, but the hierarchy is now stable enough for calculator and article cluster growth.

## Future-ready areas
- scaffolded generation of calculators and support pages through `scripts/scaffold-page.mjs`
- automatic sitemap rebuilds through `scripts/build-sitemap.mjs`
- page-type specific schema
- reviewed-date handling
- internal related-link logic generated from shared page data
- country and currency expansion later without UK-first content dilution now


## Batch 3 update
- Shared calculator behaviour now lives in `assets/js/site.js` using `data-calculator` bindings.
- Calculator pages use a reusable content shell: hero summary, form, result panel, worked examples, assumptions, practical advice, related links and FAQ.
- Event tracking readiness now includes calculator input changes and result views, in addition to internal related-link clicks.


## Final handoff note
The handoff build remains plain static HTML, CSS and JS on purpose. The current information architecture, URLs and shared patterns are stable enough to convert into a generator later, but not so repetitive that a generator is required for the initial launch-quality handoff.
